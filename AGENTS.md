# AGENTS.md

## Назначение

Этот репозиторий предназначен только для инфраструктуры деплоя.  
Он управляет оркестрацией контейнеров, reverse proxy, конфигурацией окружения и автоматизацией деплоя приложения.

Текущий контур деплоя:
- сервис `frontend`
- сервис `backend`
- `database` (PostgreSQL)
- reverse proxy `nginx`

Не реализовывать и не изменять здесь бизнес-логику.

## Базовые принципы

- Деплой должен оставаться простым, предсказуемым и воспроизводимым.
- Предпочитать минимальные и точечные правки вместо широких рефакторингов.
- Сохранять обратную совместимость текущего процесса деплоя.
- Не добавлять новые инфраструктурные компоненты без явной необходимости.

## Границы инфраструктуры

- Можно: Docker/compose, конфиг `nginx`, deploy-скрипты, CI/CD-интеграция, env-шаблоны.
- Нельзя: логика API, логика фич frontend, доменная (бизнес) логика.
- Если запрошенное изменение относится к коду приложения, перенаправлять его в соответствующие репозитории приложения.

## Правила работы с Docker и Compose

- Не ломать существующие связи сервисов в `docker-compose.yml`.
- Не менять открытые порты без явной причины.
- Сохранять `depends_on`, healthchecks, политики рестарта и ограничения ресурсов, если нет явного инцидента/требования.
- Если в Dockerfile уже используется многоступенчатая сборка (multi-stage), сохранять или улучшать этот подход.
- Безопасно оптимизировать размер образов:
  - использовать небольшие базовые образы, если это совместимо,
  - не переносить лишние build-зависимости в runtime-слои,
  - поддерживать актуальный и эффективный `.dockerignore`.

### Чеклист безопасности для Compose

Перед изменением `docker-compose.yml`:
- Проверить, что маршрутизация `nginx -> frontend/backend` остается корректной.
- Проверить, что связность `backend -> database` и env-контракт остаются корректными.
- Проверить, что healthchecks по-прежнему отражают реальную готовность сервисов.
- Проверить, что существующие volumes не удалены непреднамеренно.

### Пример безопасного изменения Docker

Хорошо:
- Добавить/скорректировать `healthcheck` для существующего сервиса.
- Уменьшить размер образа без изменения runtime-поведения.

Плохо:
- Изменить проброс порта `backend` без обновления маршрутизации.
- Удалить сервис из compose, потому что он «выглядит неиспользуемым».

## Правила работы с Nginx

- Не менять базовое проксирование без явной причины.
- Сохранять совместимость с текущими маршрутами:
  - `/` -> frontend
  - `/api/` -> backend
  - `/health` -> health backend
  - маршруты на `:8888` для Allure
- Не ломать SSL-поведение, редиректы и security headers.
- Сохранять proxy-заголовки (`Host`, `X-Real-IP`, `X-Forwarded-*`), если нет осознанного редизайна.

### Пример безопасного изменения Nginx

Хорошо:
- Точечно настроить таймауты под реальную задержку upstream.
- Добавить узконаправленный `location` для оптимизации кэширования статики.

Плохо:
- Менять префиксы маршрутов (`/api/`) без синхронного обновления backend/frontend.
- Удалять security headers, чтобы «починить» клиентские проблемы.

## Правила работы с переменными окружения

- Никогда не коммитить реальные секреты или реальные значения из `.env`.
- Использовать `.example.env` как единственный коммитимый шаблон env-контракта.
- Каждую новую переменную обязательно добавлять в `.example.env` с placeholder-значением и коротким пояснением при необходимости.
- Сохранять единообразие имен переменных между:
  - `docker-compose.yml`
  - `deploy.sh` (если используется)
  - ожиданиями runtime приложения

### Чеклист для изменений env

При добавлении новой переменной:
- Добавить ее в `.example.env`.
- Подключить ее в compose/скрипте, где это требуется.
- Убедиться, что секреты не захардкожены в файлах репозитория.

## Правила для deploy.sh

- Скрипт должен оставаться идемпотентным: повторные запуски приводят к корректному целевому состоянию деплоя.
- Не добавлять интерактивные запросы (`read`, подтверждения, меню).
- Скрипт должен оставаться CI-friendly и неинтерактивным.
- Явно логировать ключевые этапы:
  - старт,
  - обновление исходников,
  - шаги сборки/перезапуска,
  - завершение/ошибка.
- Предпочитать явное завершение при ошибках (`set -e` / надежные проверки).

### Пример безопасного изменения deploy-скрипта

Хорошо:
- Добавить понятные логи по шагам и ненулевой выход при ошибке.
- Добавить опциональные неинтерактивные флаги с разумными значениями по умолчанию.

Плохо:
- Добавлять вопросы вида `Continue? [y/N]`.
- Добавлять локальные зависимости, отсутствующие в CI/контейнерном окружении.

## Правила CI/CD (совместимость с GitHub Actions)

- Все скрипты должны выполняться в неинтерактивной CI-среде.
- Не полагаться на локальные бинарники/утилиты разработчика, если есть контейнерная альтернатива.
- Использовать детерминированные команды, подходящие для раннеров GitHub Actions.
- По возможности сохранять совместимость с branch/PR-автоматизацией.

## Стиль изменений и дисциплина ревью

- Вносить минимальные и точные изменения.
- Не переписывать файл целиком без веской причины.
- Сохранять текущую структуру проекта и принятые соглашения по именованию.
- Нестандартные инфраструктурные решения кратко документировать комментариями в конфиге/скрипте.

## Разрешенные улучшения

- Улучшение логирования и наблюдаемости деплоя.
- Упрощение и повышение безопасности процесса деплоя.
- Оптимизация Docker-образов и ускорение сборки.
- Добавление healthchecks/readiness checks.
- Повышение надежности (таймауты, ретраи, graceful restart).

## Запрещенные изменения без явного запроса

- Изменение доменов, базовых URL или публичных endpoint-ов.
- Нарушение обратной совместимости команд/процессов деплоя.
- Удаление существующих сервисов из compose.
- Радикальная смена архитектуры (например, замена ключевых компонентов) без согласования.

## Правило принятия решений для агентов

Если изменение повышает риск и явно не запрошено:
- остановиться,
- предложить минимальную альтернативу,
- запросить подтверждение перед продолжением.

Если неясно, относится ли изменение к инфраструктуре или бизнес-логике:
- считать задачу выходящей за рамки этого репозитория и запросить уточнение.
